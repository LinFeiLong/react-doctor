# React Native & Expo — Candidate Diagnostic Rules

Research synthesis of **net-new, statically-detectable** lint/diagnostic rules to expand
react-doctor's React Native and Expo coverage.

Candidates were mined from first-party docs, core repos, RFCs, optimization guides, starter
templates, testing libraries, and upgrade tooling, then **deduped** against:

- the existing **29 RN AST rules** (`packages/oxlint-plugin-react-doctor/src/plugin/rules/react-native/`),
  plus adjacent cross-cutting React rules, and
- the **11 expo-doctor project checks** added in [PR #583](https://github.com/millionco/react-doctor/pull/583)
  (`packages/core/src/checks/expo/`).

None of the candidates below duplicate PR #583. The project-level candidates (`expo-*`,
`rn-no-new-arch-incompatible-packages`) are designed to drop into the same `core/src/checks/expo`
pass that PR #583 introduces.

## Summary

**46 net-new candidate rules (+1 extension)** across 10 categories, every one statically detectable
from one of: a JS/TS/JSX **AST pattern**, **package.json**, a **config/text file**
(metro/babel/app config, tsconfig, eas.json, `.gitignore`, `.env*`), or **filesystem/git** state.

react-doctor does **not** execute code, hit the network, query the Expo API, run the Expo/RN CLI, or
inspect native iOS/Android projects — so anything requiring those is intentionally excluded (see
[Considered but deliberately excluded](#considered-but-deliberately-excluded)).

## How to read the tables

- **Prefix** = surface. `rn-*` are mostly oxlint AST rules over JS/TS source; `expo-*` (and a few
  `rn-*`) are project-level checks over `package.json` / config / git, like PR #583's checks.
- **T** = priority tier. **T1** = build first (high value, low noise); **T2** = solid; **T3** =
  hygiene / borderline.
- Columns: rule id, description (catches X → Y), justification + source, detection (signal + key
  false-positive trap), fix.

## Cheap win: extend an existing rule

`rn-no-deprecated-modules` already maps removed core modules; just add the **PropTypes removed from
core in RN 0.66** (`ViewPropTypes`, `ColorPropType`, `PointPropType`, `EdgeInsetsPropType`,
`ImagePropTypes`, `TextPropTypes`) plus `ToolbarAndroid` / `CheckBox` / `SegmentedControlIOS` to
`DEPRECATED_RN_MODULE_REPLACEMENTS` (`packages/oxlint-plugin-react-doctor/src/plugin/constants/react-native.ts`).
Same detection surface (named import from `react-native`), near-zero effort.

## Considered but deliberately excluded

- `**rn-react-compiler-not-enabled`** / `**rn-android-enable-r8\*\*`— detectable but opinionated and
high-noise (fire on nearly every project; R8 also needs native`build.gradle` outside scope).
  Opt-in at most.
- `**useNativeDriver:false**` (`rn-animated-no-js-driver`) — 3 agents converged on excluding:
  overlaps `rn-prefer-reanimated`, and layout/color animations legitimately require `false` →
  unavoidable false positives.
- `**rn-no-busy-wait-loop**` — real (`while (Date.now()-start < N)`) but far too rare to justify.
- **Already covered** (confirmed via grep): barrel imports (`no-barrel-import`), full-lodash
  (`no-full-lodash-import`), moment (`no-moment`), Intl-formatter hoisting (`js-hoist-intl`),
  stack/bottom-tabs navigators (`rn-no-non-native-navigator`), legacy shadow → `boxShadow`
  (`rn-style-prefer-boxshadow`, = eslint-plugin-expo `prefer-box-shadow`), `SafeAreaView` (in
  `rn-no-deprecated-modules`), static inline objects/styles (`prefer-module-scope-static-value` +
  `no-inline-prop-on-memo-component`), manual memo (`react-compiler-no-manual-memoization`).
- **Not statically detectable** (need runtime / native / network): whether New Arch is actually
  enabled (native `gradle.properties`), third-party New-Arch native compatibility, view flattening
  (`collapsable`), SSL pinning, OAuth/PKCE correctness, FlatList `windowSize`/`getItemLayout` tuning.
  The curated lists (awesome-react-native, React-Native-Apps, open-source-react-native-apps) and the
  Futurice repo (an empty stub) yielded zero rules — they are discovery indexes only.
